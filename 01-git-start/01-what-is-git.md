# Git이란?

> 버전 관리의 개념, Git의 탄생과 철학, 분산 버전 관리

## 개요

코드를 작성하다 보면 "어제 버전이 더 나았는데..."라는 순간이 반드시 옵니다. Git은 바로 그런 순간을 위한 도구입니다. 이 섹션에서는 Git이 무엇이고, 왜 전 세계 개발자들의 필수 도구가 되었는지 알아봅니다.

**선수 지식**: 터미널(명령 프롬프트)을 열어본 경험이 있으면 좋지만, 없어도 괜찮습니다.
**학습 목표**:
- 버전 관리 시스템(VCS)이 무엇인지 이해한다
- Git의 핵심 철학과 특징을 설명할 수 있다
- 중앙집중식과 분산형 버전 관리의 차이를 구분할 수 있다

## 왜 알아야 할까?

혹시 파일을 이렇게 관리한 적 있나요?

- `보고서_최종.docx`
- `보고서_최종_수정.docx`
- `보고서_진짜최종.docx`
- `보고서_진짜최종_v2_이걸로제출.docx`

누구나 한 번쯤은 겪는 상황이죠. 이런 방식은 혼자 작업할 때도 혼란스러운데, 여러 명이 함께 작업한다면 어떨까요? Git은 이 혼란을 깔끔하게 해결해 줍니다. 오늘날 **소프트웨어 개발자의 약 97%가 Git을 사용**하고 있으며, 개발자뿐 아니라 디자이너, 작가, 데이터 과학자까지 Git의 힘을 빌리고 있습니다.

## 핵심 개념

### 버전 관리 시스템(VCS)이란?

> 💡 **비유**: 버전 관리는 **게임의 세이브 시스템**과 같습니다. 게임을 진행하면서 중요한 순간마다 저장해두면, 실수했을 때 언제든 그 시점으로 돌아갈 수 있죠. 코드도 마찬가지입니다.

**버전 관리 시스템(Version Control System, VCS)**은 파일의 변경 이력을 체계적으로 기록하고 관리하는 도구입니다. 핵심 기능은 다음과 같습니다:

- **변경 추적**: 누가, 언제, 무엇을 바꿨는지 기록
- **되돌리기**: 과거 어느 시점으로든 복원 가능
- **협업**: 여러 사람이 동시에 같은 프로젝트를 작업
- **분기**: 실험적인 시도를 안전하게 진행

### Git은 어떤 VCS인가?

> 💡 **비유**: 기존의 버전 관리가 **도서관에 한 권뿐인 원본 책**이라면, Git은 **모든 사람이 원본 전체를 복사해서 갖고 있는 시스템**입니다. 도서관이 문을 닫아도 누구든 전체 기록을 가지고 작업할 수 있죠.

Git은 **분산 버전 관리 시스템(DVCS)**입니다. 이전 세대의 VCS와 비교해 볼까요?

| 특성 | 중앙집중식 (SVN 등) | 분산형 (Git) |
|------|---------------------|-------------|
| 저장소 위치 | 서버에만 전체 이력 | 모든 개발자가 전체 이력 보유 |
| 오프라인 작업 | 불가능 | 가능 (거의 모든 작업) |
| 속도 | 네트워크 의존 | 로컬이라 매우 빠름 |
| 서버 장애 시 | 작업 중단 | 영향 없음 |
| 브랜치 비용 | 무거움 (디렉토리 복사) | 가벼움 (포인터 하나) |

Git의 핵심 특징을 정리하면:

- **빠름**: 대부분의 작업이 로컬에서 이루어져 네트워크 지연이 없습니다
- **무결성**: 모든 데이터를 체크섬(SHA-1 해시)으로 관리하여 변조를 감지합니다
- **스냅샷 방식**: 파일의 차이(diff)가 아닌, 전체 스냅샷을 저장합니다
- **브랜치가 가벼움**: 새로운 브랜치를 만드는 데 0.01초도 안 걸립니다

### Git의 동작 원리 — 스냅샷

다른 VCS(SVN, Perforce 등)가 "파일별 변경 사항(delta)"을 저장하는 것과 달리, Git은 **프로젝트 전체의 스냅샷**을 저장합니다. 변경되지 않은 파일은 이전 스냅샷의 링크만 저장하므로 공간 낭비도 없습니다.

> ⚠️ **흔한 오해**: "Git은 변경된 부분만 저장한다" — 사실 Git은 **스냅샷 기반**입니다. 매 커밋마다 프로젝트 전체 상태를 기록하되, 변하지 않은 파일은 이전 스냅샷에 대한 링크로 효율적으로 관리합니다.

## 실습: Git 버전 확인하기

Git이 이미 설치되어 있다면, 현재 버전을 확인해 보세요:

```bash
# Git 버전 확인
git --version
```

```output
git version 2.48.0
```

아직 설치되지 않았다면 걱정하지 마세요. 다음 섹션에서 설치 방법을 자세히 다룹니다.

```bash
# Git이 제공하는 명령어 목록 둘러보기
git help -a
```

## 더 깊이 알아보기

### Git의 탄생 — 2주 만에 만든 혁명

2005년 4월, **Linus Torvalds**는 큰 문제에 직면했습니다. 리눅스 커널 개발에 사용하던 상용 버전 관리 도구 **BitKeeper**의 무료 라이선스가 취소된 것이죠. Andrew Tridgell이 BitKeeper의 프로토콜을 리버스 엔지니어링한 것이 원인이었습니다.

Torvalds는 기존의 어떤 VCS도 리눅스 커널 개발에 적합하지 않다고 판단하고, 직접 새로운 도구를 만들기로 결심합니다. 그의 설계 목표는 명확했습니다:

- **빠를 것** — 수천 명의 개발자가 참여하는 프로젝트를 감당할 수 있어야 함
- **분산형일 것** — 중앙 서버 없이도 완전하게 동작해야 함
- **안전할 것** — 데이터 손상이나 변조를 감지할 수 있어야 함

놀랍게도 Torvalds는 **2005년 4월 3일에 개발을 시작해, 약 2주 만에** Git의 첫 버전을 완성했습니다. 같은 해 7월에 Junio C Hamano에게 유지보수를 넘겼고, 12월에 1.0이 릴리스되었습니다.

> 💡 **알고 계셨나요?**: "Git"이라는 이름은 영국 속어로 "고집불통, 멍청이"라는 뜻인데요. Linus Torvalds가 자기 자신을 빗대어 지은 이름이라고 합니다. Git 공식 README에는 "the stupid content tracker"라고 적혀 있죠.

### Git의 현재와 미래

2025년 기준 Git의 최신 안정 버전은 **2.48**입니다. 주요 최근 동향을 살펴보면:

- **SHA-256 전환 진행 중**: Git은 SHA-1에서 SHA-256으로의 전환을 준비하고 있습니다. Git 3.0에서 SHA-256이 기본이 될 예정이며, 2026년 말 릴리스를 목표로 하고 있습니다
- **reftable 백엔드**: 레퍼런스(브랜치, 태그) 저장 방식의 현대화가 진행 중입니다
- **메모리 누수 해결**: Git 2.48에서 테스트 스위트 기준 메모리 누수가 완전히 해결되었습니다

## 흔한 오해와 팁

> ⚠️ **흔한 오해**: "Git과 GitHub는 같은 것이다" — Git은 **버전 관리 도구**이고, GitHub는 Git 저장소를 **호스팅하는 웹 서비스**입니다. Git 없이 GitHub를 쓸 수 없지만, GitHub 없이 Git은 완벽하게 동작합니다. GitLab, Bitbucket 등 다른 호스팅 서비스도 있습니다.

> 🔥 **실무 팁**: Git은 텍스트 파일에 최적화되어 있습니다. 바이너리 파일(이미지, 동영상 등)도 관리할 수 있지만, 변경 이력이 비효율적으로 커질 수 있어요. 대용량 바이너리는 [Git LFS](../12-troubleshooting/02-large-repos.md)를 고려하세요.

## 핵심 정리

| 개념 | 설명 |
|------|------|
| VCS | 파일의 변경 이력을 추적하고 관리하는 시스템 |
| Git | 분산 버전 관리 시스템(DVCS), 2005년 Linus Torvalds가 개발 |
| 분산형 | 모든 개발자가 전체 이력을 로컬에 보유 |
| 스냅샷 | Git은 파일의 차이가 아닌, 프로젝트 전체 상태를 저장 |
| SHA 해시 | 모든 데이터의 무결성을 체크섬으로 보장 |

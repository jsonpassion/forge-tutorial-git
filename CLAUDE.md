## 콘텐츠 작성 규칙 (가장 중요!)

### 톤 & 스타일
- **언어**: 한국어 (기술 용어는 영문 병기. 예: "브랜치(branch)", "커밋(commit)", "리베이스(rebase)")
- **화자**: "~합니다" 존댓말 사용, 친근하지만 전문적인 톤
- **대상 독자**: 프로그래밍 기초는 있지만 Git은 처음이거나 기초만 아는 개발자
- **핵심 원칙**: "초등학생도 비유로 이해할 수 있게, 하지만 시니어 개발자도 깊이에 만족할 수 있게"
- **어투 가이드**: 딱딱한 교과서체가 아닌, **옆에서 설명해주는 선배** 느낌으로 작성
  - "~입니다" 일변도 대신 "~인데요", "~거든요", "~죠" 등 자연스러운 종결어미 적절히 혼용
  - 독자에게 질문을 던지는 구문 활용: "왜 그럴까요?", "어떤 차이가 있을까요?"
  - 놀라운 사실이나 반전을 만나면 감탄/공감 표현: "놀랍게도", "사실은", "의외로"
  - 단, 과도한 구어체나 반말은 금지. 존댓말 기반의 친근한 톤 유지

### 비유와 예시 규칙
- 모든 새로운 개념에는 반드시 **일상적 비유**를 먼저 제시
- 비유 → 실제 개념 → 명령어 순서로 설명
- 비유 예시:
  - 커밋 → "게임 세이브 포인트: 언제든 이 시점으로 돌아올 수 있습니다"
  - 브랜치 → "평행 세계: 메인 스토리에 영향 없이 새로운 시도를 해볼 수 있는 공간"
  - 스테이징 → "무대 위 리허설: 확정 전에 무엇을 올릴지 미리 골라두는 과정"
  - merge → "두 갈래 길이 다시 하나로 합쳐지는 합류 지점"
  - rebase → "기차 노선 재배치: 갈라졌던 선로를 깔끔하게 일직선으로 정리하는 것"
  - clone → "건물 전체를 복사해서 내 땅에 똑같이 짓는 것"
  - fork → "원본 레시피를 가져와서 내 입맛대로 변형하는 것"
  - stash → "작업 중인 책상을 서랍에 임시로 넣어두고, 깨끗한 책상에서 다른 일을 하는 것"
  - HEAD → "지금 내가 서 있는 위치를 가리키는 화살표"
  - .gitignore → "택배 기사에게 '이 방은 무시해주세요'라고 메모 붙여두기"
  - Pull Request → "동료에게 '제 작업 좀 확인해주세요'라고 검토 요청서를 보내는 것"
  - conflict → "같은 문장을 두 사람이 동시에 다르게 고쳤을 때, 어떤 버전을 쓸지 결정하는 것"

### 스토리와 맥락 규칙
- 각 섹션에 해당 기술/개념의 **역사적 에피소드나 탄생 배경**을 1~2개 포함
- 발명자의 이야기, 이름의 유래, 의외의 탄생 경위 등 독자가 "오!" 하는 이야기
- "더 깊이 알아보기" 또는 본문 중 자연스러운 위치에 배치
- 예시:
  - "Git은 2005년 Linus Torvalds가 리눅스 커널 개발을 위해 단 2주 만에 만들었습니다"
  - "'Git'이라는 이름은 영국 속어로 '고집불통'이라는 뜻인데, Linus가 자기 자신을 빗대어 지었다고 합니다"
  - "GitHub는 2008년 Tom Preston-Werner가 만들어 2018년 Microsoft에 75억 달러에 인수되었습니다"
  - "브랜치 개념은 사실 소프트웨어 버전 관리에서 1970년대부터 존재했지만, Git이 이를 혁명적으로 가볍게 만들었습니다"

### 흔한 오해와 팁 규칙
- 각 섹션에 **> ⚠️ 흔한 오해** 또는 **> 💡 알고 계셨나요?** 박스를 최소 1개 포함
- 초보자가 자주 혼동하는 개념, 흔히 하는 실수, 직관과 다른 사실 등
- 형식: 인용 블록(`>`) + 이모지 헤더로 시각적으로 구분
- 예시:
  - `> ⚠️ **흔한 오해**: "git pull은 안전하다" — 사실 pull은 fetch + merge이므로, 예상치 못한 머지 커밋이 생길 수 있습니다. fetch 후 확인하는 습관을 들이세요.`
  - `> 💡 **알고 계셨나요?**: git의 커밋 해시는 SHA-1이지만, Git은 현재 SHA-256으로의 전환을 준비하고 있습니다.`
  - `> 🔥 **실무 팁**: force push(--force)는 절대 공유 브랜치에 하지 마세요. 불가피하면 --force-with-lease를 쓰세요.`
  - `> ⚠️ **흔한 오해**: "rebase가 merge보다 항상 좋다" — 상황에 따라 다릅니다. 공유 브랜치에서는 merge가, 개인 브랜치 정리에는 rebase가 적합합니다.`

### 코드 블록 vs 마크다운 구분 (매우 중요!)

앱에서 코드 블록(` ``` `)은 "code" 라벨이 붙고 코드 전용 스타일로 렌더링됩니다.
**실행 가능한 명령어만 코드 블록에 넣고, 다이어그램/시각적 설명은 반드시 일반 마크다운으로 작성합니다.**

- **코드 블록 사용 O**: Git/bash 명령어, YAML 파일, 설정 파일, 터미널 출력
- **코드 블록 사용 X**: 브랜치 다이어그램, 커밋 그래프, 워크플로우 도식, 아키텍처 구조도

다이어그램 대체 방법:
- **테이블**: 비교표 (merge vs rebase), 명령어 옵션 비교, 상태 설명
- **인용(`>`)**: 부연 설명, 흐름 요약, 팁/주의사항
- **볼드/이모지**: 흐름 표시 (→, ←, ↕️), 상태 표시, 브랜치 관계
- **리스트**: 순서가 있는 흐름/파이프라인, 워크플로우 단계

### 특수 코드 블록 태그

ForgeCore의 CodeBlockView는 기본 구문 하이라이팅 외에 특수 언어 태그를 지원합니다.
명령어 실행 결과, 에러 메시지, 터미널 출력을 시각적으로 구분하여 표시할 수 있습니다.

| 태그 | 용도 | 스타일 |
|------|------|--------|
| `output`, `result` | 명령어 실행 결과 | 회색 배경, 모노스페이스 |
| `error` | 에러 메시지 | 빨간 틴트 배경 |
| `console`, `terminal` | 터미널 출력/프롬프트 | 다크 배경, 초록 텍스트 |
| `run:{언어}` | 인터랙티브 실행 시뮬레이션 | 실행 버튼 + 출력 애니메이션 |

사용 예시:

````markdown
```bash
git log --oneline -5
```

```output
a1b2c3d (HEAD -> main) Add README
e4f5g6h Initial commit
```

```error
error: failed to push some refs to 'origin/main'
hint: Updates were rejected because the remote contains work that you do not have locally.
```

```console
$ git status
On branch main
Your branch is up to date with 'origin/main'.

nothing to commit, working tree clean
```
````

#### 인터랙티브 실행 블록 (`run:`)

`run:{언어}` 태그를 사용하면 사용자가 "실행" 버튼을 눌러 코드 실행을 시뮬레이션할 수 있습니다.
바로 다음에 `output` 블록을 배치하면 예상 출력으로 자동 페어링됩니다.

````markdown
```run:bash
echo "현재 브랜치:"
git branch --show-current
```

```output
현재 브랜치:
main
```
````

**사용 규칙:**
- `run:` 직후에 `output` 블록을 반드시 배치 (출력이 있는 코드의 경우)
- `echo` 등 텍스트 출력 명령만 사용 (GUI 명령 제외)
- 한 섹션에 1~3개 적정 (너무 많으면 학습 흐름 방해)
- 출력이 명령 실행 결과와 정확히 일치해야 함

- **명령어 실행 결과**는 주석(`#`) 대신 `output` 또는 `result` 블록으로 분리하여 가독성 향상
- **에러 예시**는 `error` 블록으로 시각적으로 경고 표시
- **터미널 프롬프트 포함 출력**은 `console` 또는 `terminal` 블록 사용
- 간단한 한 줄 결과는 기존처럼 주석(`# → a1b2c3d`)으로 표시해도 무방

### 코드 예제 규칙
- **언어**: bash/shell 기준 (Git 명령어)
- **크로스 플랫폼**: macOS/Windows/Linux 차이가 있는 경우 모두 표기
- **실행 가능**: 터미널에서 복사-붙여넣기로 바로 실행할 수 있어야 함
- **주석**: 핵심 라인마다 한국어 주석 필수
- **출력 포함**: 명령어 실행 결과를 주석이나 별도 블록으로 표시
- **점진적**: 간단한 예제 → 심화 예제 순서
- **위험 표시**: 파괴적 명령어(`--force`, `reset --hard` 등)에는 경고 주석

```bash
# 좋은 예시 — 단계별 실습
# 1. 새 저장소 만들기
mkdir my-project && cd my-project
git init

# 2. 파일 만들고 스테이징
echo "# My Project" > README.md
git add README.md

# 3. 첫 커밋
git commit -m "Initial commit"
```

```output
Initialized empty Git repository in /Users/you/my-project/.git/
[main (root-commit) a1b2c3d] Initial commit
 1 file changed, 1 insertion(+)
 create mode 100644 README.md
```

```bash
# ⚠️ 주의: 이 명령어는 되돌릴 수 없습니다!
git reset --hard HEAD~1
```

## 이미지 규칙

### 저장 및 명명
- 경로: `assets/images/` 디렉토리에 저장
- 형식: PNG 권장 (스크린샷, 다이어그램), GIF (터미널 애니메이션)
- 명명: `{챕터번호}-{섹션번호}-{설명}.png` (예: `03-03-merge-diagram.png`)
- **다이어그램은 코드 블록이 아닌 마크다운(테이블, 인용, 리스트)으로 표현**
- 터미널 스크린샷은 다크 테마 기준으로 캡처

### 경로 규칙
ForgeCore가 이미지 경로를 자동 처리합니다:
- **상대경로 사용**: `![alt](images/ch03/merge.png)` → baseURL에 자동 결합
- **파일명만 사용**: `![alt](merge.png)` → `assets/images/merge.png`으로 자동 변환 (레거시 호환)
- **권장 형식**: `![브랜치 병합 다이어그램](assets/images/03-03-merge-diagram.png)`

### 출처 표기
외부 이미지(Git 공식 사이트, GitHub Docs, 블로그 등)를 사용할 때 출처를 표기합니다.
ForgeCore가 마크다운 이미지 title 필드에서 출처를 자동 추출하여 표시합니다.

| 형식 | 마크다운 | 앱 표시 |
|------|----------|---------|
| 출처 없음 | `![설명](path)` | 출처 표시 안 함 |
| URL만 | `![설명](path "https://source-url.com")` | 도메인 자동 추출 |
| 커스텀 라벨 | `![설명](path "Pro Git \| https://git-scm.com/book")` | 커스텀 라벨 표시 |

### 이미지 뷰어 동작 (ForgeCore 내장)
- **사이즈**: 원본 초과 확대 방지, maxHeight 500pt 제한
- **프레임**: 테두리 + 미세 그림자
- **확대**: macOS(클릭→sheet), iOS(탭→풀스크린 핀치줌/더블탭/스와이프닫기)
- **대용량**: 5MB 초과 시 CGImageSource 다운샘플링(2048px)

## manifest.json 형식
```json
{
  "chapters": [
    {
      "id": "chapter-id",
      "title": "챕터 제목",
      "description": "설명",
      "sections": [
        {
          "id": "section-id",
          "title": "섹션 제목",
          "file": "폴더/파일명.md",
          "description": "설명"
        }
      ]
    }
  ]
}
```
manifest.json 구조는 절대 임의로 변경하지 않습니다. 변경이 필요하면 사용자에게 먼저 확인합니다.

### manifest.json 버전 규칙 (Semantic Versioning)
manifest.json의 `version` 필드는 다음 규칙에 따라 관리합니다:

| 버전 변경 | 조건 | 예시 |
|-----------|------|------|
| `1.0.X` → `1.0.(X+1)` | 콘텐츠 수정/추가 시 (섹션 작성, 오타 수정 등) | 1.0.7 → 1.0.8 |
| `1.X.0` → `1.(X+1).0` | 챕터 추가/구조 변경 시 (manifest 구조 변경) | 1.1.0 → 1.2.0 |
| `X.0.0` → `(X+1).0.0` | 대규모 리뉴얼 시 (전체 재구성, 기술 스택 변경 등) | 1.0.0 → 2.0.0 |

- 상위 버전이 올라가면 하위 버전은 0으로 리셋 (예: 1.2.3 → 1.3.0)
- **콘텐츠 커밋 시 반드시 patch 버전(X) 증가**
- `lastUpdated` 날짜도 함께 갱신

## 중요 주의사항
1. **순서 준수**: 반드시 manifest.json의 챕터/섹션 순서대로 작성
2. **일관성**: 모든 섹션이 동일한 구조와 톤을 유지
3. **웹 검색 필수**: 콘텐츠 작성 전 반드시 해당 주제의 최신 자료를 웹 검색
4. **status 업데이트 필수**: 섹션 완료 시 contents_status.md 반드시 갱신
5. **참고 자료 필수**: 모든 섹션에 유효한 외부 링크 포함 (Pro Git, Git 공식 문서 우선)
6. **코드 블록 구분**: 실행 명령어만 코드 블록, 다이어그램/구조 설명은 마크다운 테이블/인용 사용
7. **스토리 필수**: 모든 섹션에 역사적 에피소드, 탄생 배경, Git/GitHub 관련 이야기 등 최소 1개 포함
8. **오해/팁 필수**: 모든 섹션에 흔한 오해, 혼동되는 개념, 실무 팁 중 최소 1개 포함
9. **친근한 어투**: 교과서체를 지양하고, 옆에서 설명해주는 선배 느낌의 자연스러운 톤 유지
10. **위험 표시**: `--force`, `reset --hard`, `clean -fd` 등 파괴적 명령어에는 반드시 경고 추가
11. **크로스 플랫폼**: macOS/Windows/Linux 간 차이가 있는 명령어는 모두 표기
12. **개인정보 처리방침 준수**: 이 앱은 **개인정보를 수집하지 않으며**, 외부 분석 SDK/광고를 사용하지 않고, 로컬 저장만 사용함
13. **앱 이름 통일**: 앱을 지칭할 때는 **Forge: Git**, 튜토리얼 제목은 **2026: Git 완전 정복**으로 표기
